## Runtime environment (important)

Your current working directory **is** the live runtime workspace. There is no "source repo vs. running copy" split — the files you edit are the same ones the `9501` and `9502` processes are running. After your edits, trigger a reload and they take effect immediately.

Layout:

- `gui/` — frontend source (rebuild after changes)
- `gui/dist/` — frontend build output
- `server/main/` — the platform itself; exposes APIs that any app can call
- `server/apps/` — apps that have their own backend, one directory per app
- `server/shared/` — shared backend code
- `apps/<appname>/APP.md` — runtime per-app documentation for the active locale (gitignored; baked from `language/<locale>/apps/<appname>/APP.md`)
- `language/` — multi-language source assets (AGENTS.md / CLAUDE.md / per-app docs / UI strings; edit sources here, runtime is generated by the bake)
- `database/` — SQLite files
- `files/` — workspace files

## App model

iimos has no "system app vs user app" distinction — **every entry in the rail is an app, all peers**. The only difference is on the backend side:

- **Uses platform APIs**: Chat / Tasks / Settings are core apps; they consume the APIs and ws exposed by `server/main/`
- **Has a custom backend**: Todo lives in `server/apps/<name>/` with its own table and routes
- **Pure frontend**: none today, but possible (e.g. small local utilities)

Platform capabilities (any app can curl these):

| Capability | Endpoint |
|--|--|
| Chat / Agent | `/api/chat/*` + `/ws` |
| Tasks | `POST /api/task/create/instant` `POST /api/task/create/agent` `GET /api/task` |
| Settings | `/api/settings` |
| Runtime | `/api/runtime/*` |

## App development

Before creating or modifying an app:

1. Follow the app development rules in this prompt
2. To reference an existing app, read `apps/<appname>/APP.md` (the baked runtime doc) plus `server/apps/<appname>/` and `gui/src/apps/<appname>/`. **To edit the doc, edit the source** at `language/<locale>/apps/<appname>/APP.md`
3. Register the app in three places: `server/apps/registry.js` (if it has a backend) + `gui/src/apps.js` + `gui/src/stores/view.js` (rail nav — see grouping rule below)

### Sidebar grouping (architectural convention)

`gui/src/stores/view.js` exports `navGroups`, which partitions the rail into three semantically distinct sections:

- `navGroups.top` — fixed top section; only Chat and Tasks go here (core operation entry points, never scrolls)
- `navGroups.apps` — scrollable middle section, labelled "应用" / "Apps"; **new apps go here**; auto-scrolls when content exceeds the available height
- `navGroups.bottom` — fixed bottom section; only Settings goes here; the theme toggle is rendered separately

The theme toggle is rendered separately in AppDrawer above the bottom group and is **not part of any navGroup**.

To register a new app = append `{ path, label, msi }` to `navGroups.apps`. **Do not add to top or bottom** — those are reserved core entry sections; mixing apps in breaks the grouping semantics.

### Calling auth-protected endpoints

`/api/*`, `/apps/*`, `/files/*`, `/ws*` are all auth-protected. As an in-process agent you have a dedicated API token, injected via the `IIMOS_API_TOKEN` environment variable (loaded from the DB at server startup).

**Always reference the variable name, never inline the raw value:**

```
curl -H "Authorization: Bearer $IIMOS_API_TOKEN" http://localhost:9501/api/settings
```

Don't do this:

- Don't `echo $IIMOS_API_TOKEN` or `env | grep TOKEN` and put the value into the conversation
- Don't copy-paste the raw token into any command
- Don't write the token into files, memories, logs, or commit messages

The system applies token redaction (replacing the raw value back with the literal `$IIMOS_API_TOKEN`) before DB writes / WebSocket pushes / log output, but that's a safety net, not absolution.

## Reload (hard rule)

**All backend code changes require a reload request, otherwise nothing takes effect.**

The `9501` (main) and `9502` (apps) Node processes bake the module tree into memory at startup. When you edit a `.js` file on disk, **the in-memory copy is not refreshed** — Node's ESM module cache does not hot-reload on file change. No reload = no effect.

**Right after your edit, call:**

```bash
curl -X POST http://localhost:9501/api/runtime/reload/request \
  -H "Content-Type: application/json" \
  -d '{"build": false, "restartApps": true, "restartServer": false, "message": "added todo app"}'
```

Parameters:

- `build: true` → you edited anything under `gui/` (`.vue` / `.ts` / `.js` / `.css` / Tailwind)
- `restartApps: true` → you edited anything under `server/apps/` (**including `registry.js`**)
- `restartServer: true` → you edited `server/main/`, `server/shared/`

Combine flags as needed.

**Flow**: `/api/runtime/reload/request` first runs a background preflight (it spins up a new process to verify it can boot). Once preflight passes, the frontend pops up a "Restart system" dialog asking the user to confirm. Only on confirm does the actual swap happen. If preflight fails, the call errors out and the live service is untouched — so this endpoint is safe to call.

**Preflight only, no swap**:

```bash
curl -X POST http://localhost:9501/api/runtime/reload/test \
  -H "Content-Type: application/json" \
  -d '{"restartApps": true}'
```

**Forbidden**:

- Telling the user "done" without calling reload — the new code isn't running, the user will see 404 or the old behavior
- Calling `/api/runtime/reload` (the terminal endpoint) directly, bypassing preflight and user confirmation
- Setting `restartServer: true` without explicit user request — restarting main interrupts running tasks
