# AIOS 部署安全手册（面向服务器部署）

本文用于生产或公网环境部署 AIOS 时的最小安全基线。

## 1. 先明确风险

如果你的 AIOS 部署在公网服务器上：
- 访问地址一旦发给陌生人，对方就能直接探测你的服务。
- 服务器 IP 一旦暴露，可能遭受端口扫描、爆破、漏洞探测、DDoS。
- 若你在系统中配置了模型密钥或其它敏感信息，被入侵后会造成直接损失。

结论：
- 仅把访问地址发给可信的人。
- 默认按“公网可被持续扫描”的前提部署，不要按“没人会发现”来设计。

## 2. 最小部署原则

- 只开放必要端口：`80/443`。
- 不要把内部端口（如 `9700/9701`）直接暴露到公网。
- 不要把数据库目录、日志目录、代码目录做公网文件共享。
- 使用独立服务器部署 AIOS，不与高敏业务混部。
- 系统和依赖保持更新，避免长期停留在旧版本。

## 3. Cloudflare 基础防护（推荐）

目标：用户访问 Cloudflare 域名，由 Cloudflare 反向代理到你的源站；源站 IP 不直接暴露给普通访客。

### 3.1 域名接入 Cloudflare

1. 把域名 NS 切到 Cloudflare。
2. 在 DNS 中添加 `A` 记录指向你的服务器公网 IP。
3. 打开橙色云（`Proxied`），让流量经过 Cloudflare。

### 3.2 SSL/TLS

1. Cloudflare `SSL/TLS` 模式使用 `Full (strict)`。
2. 源站安装有效证书（推荐 Cloudflare Origin Certificate 或 Let's Encrypt）。
3. 打开 `Always Use HTTPS`。
4. 打开 `Automatic HTTPS Rewrites`。

### 3.3 WAF 与基础防刷

在 Cloudflare 控制台启用：
- WAF（托管规则）
- Bot Fight Mode（或等价 Bot 防护）
- Rate Limiting（对登录、任务创建、关键 API 做限流）

建议的起步限流策略：
- `/api/auth/*`：按 IP 严格限速（防爆破）
- `/api/chat/*`、`/api/task/*`：中等限速（防滥用）
- 异常高频请求直接阻断或挑战（Managed Challenge）

### 3.4 源站保护

- 在服务器防火墙中仅允许 Cloudflare 回源 IP 段访问 80/443。
- 其它来源直接拒绝，避免绕过 Cloudflare 直连源站。

## 4. 域名与 HTTPS 配置（Nginx）

示例思路：
1. 域名：`aios.example.com`。
2. Nginx 监听 `443`，绑定证书。
3. `location /` 反代到 `127.0.0.1:9700`。
4. `80` 端口全部 301 跳转到 `https`。

关键点：
- 证书必须覆盖你的实际访问域名。
- 证书到期前自动续期。
- 回源只走本机回环地址，应用端口不直接暴露。

## 5. 账户与访问控制

- 首次初始化后立即设置强密码。
- 管理员账户禁止弱密码、禁止复用其它站点密码。
- 不共享管理员账号给多人。
- 有成员离开后立即改密。

## 6. AI 与任务安全建议

- 对外部输入保持保守：不要把未知来源文本直接当高权限执行指令。
- 对“写文件、执行命令、联网下载”的任务保持人工确认习惯。
- 定期检查任务历史，发现异常任务立即停用并排查。

## 7. 日常巡检清单（建议每周）

- 检查是否仍只开放 `80/443`。
- 检查 Cloudflare 代理是否仍开启（橙云）。
- 检查 WAF/限流是否生效。
- 检查 HTTPS 证书有效期。
- 检查服务器登录日志与异常请求峰值。
- 检查 AIOS 版本并更新。

## 8. 发生异常时

如果怀疑被攻击或被入侵：
1. 立即限制外部访问（WAF 全挑战或临时下线域名）。
2. 立即更换管理员密码与相关密钥。
3. 备份日志与关键数据后再做修复。
4. 升级系统与依赖，修补暴露点后再恢复服务。
